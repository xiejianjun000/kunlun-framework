## 🌌 总纲：安全之道

> 道可道，非常道。安可安，非常安。

安全不是一个可以一劳永逸解决的问题，而是一个持续的平衡过程。就像太极中的阴阳，永远在运动，永远在转化。

OpenTaiji 的安全哲学：
- **不治已病治未病**：预防优先，在问题发生前就消除隐患
- **阴阳平衡**：安全与便利不是对立，是需要找到的平衡点
- **层层设防**：没有绝对安全的单点，但有牢不可破的纵深
- **顺势而为**：安全不应该阻碍进化，而应该护航进化

## 🐲 四象四层 · 内固其精

四象四层是 OpenTaiji 的内部安全基石，保护运行时内核的绝对安全。

---

### 第一层 🌌 青龙 · 木之道 — Actor 隔离层

> 木曰曲直，生生不息。独立不倚，互不侵犯。

**安全哲学**：每个 Actor 都是独立的生命个体，有自己的边界和领地。就像森林中的每一棵树，独立生长但共同组成生态系统。

**工程实现**：

| 原则 | 说明 | 实现方式 |
|------|------|---------|
| **内存隔离** | Actor 之间无共享内存 | 所有状态封装在 Actor 内部，通过消息传递通信 |
| **邮箱边界** | 消息是唯一的交互方式 | 禁止直接方法调用，所有交互必须通过消息邮箱 |
| **崩溃隔离** | 一个 Actor 崩溃不影响其他 | 监督树机制，崩溃隔离，分级重启 |
| **生命周期独立** | 每个 Actor 有自己的生命周期 | 独立启动、停止、重启，不依赖其他 Actor |

**核心信条**：*如果一个 Actor 被入侵，最多只能毁掉它自己。*

---

### 第二层 🦅 朱雀 · 火之道 — 内容安全层

> 火曰炎上，明察秋毫。善恶立辨，无所遁形。

**安全哲学**：就像火焰燃烧杂质，内容安全层要识别并净化所有危险的内容输出。

**工程实现**：

| 原则 | 说明 | 实现方式 |
|------|------|---------|
| **输出过滤** | 所有输出必须经过安检 | WFGY 幻觉检测 + 敏感词过滤 + 内容分级 |
| **溯源可查** | 每个输出都能追溯来源 | SourceTracer 知识溯源，输出签名机制 |
| **置信度标注** | 不确定的内容必须明确标注 | 幻觉风险分数，置信度等级展示 |
| **危险操作拦截** | 高风险操作二次确认 | 删除、修改、发送等操作需要确认机制 |

**核心信条**：*宁可少说，不可说错；如果不确定，一定要说自己不确定。*

---

### 第三层 🐢 玄武 · 水之道 — 运行时保护层

> 水曰润下，随物赋形。灵活应变，无孔不入。

**安全哲学**：就像水渗透每一个缝隙，运行时保护要覆盖执行路径的每一个环节。

**工程实现**：

| 原则 | 说明 | 实现方式 |
|------|------|---------|
| **沙箱执行** | 所有用户代码在沙箱中运行 | VM 隔离，权限最小化，资源限制 |
| **系统调用拦截** | 禁止未授权的系统调用 | Syscall 白名单，危险调用自动拦截 |
| **资源配额** | 每个租户有明确的资源上限 | CPU、内存、Token、调用次数硬限制 |
| **执行超时** | 所有操作必须有超时机制 | 死循环检测，超时自动终止 |
| **无限循环检测** | 智能检测死循环 | 执行步数计数，超过阈值自动中断 |

**核心信条**：*永远假设用户输入是恶意的，永远假设代码会出错。*

---

### 第四层 🐯 白虎 · 金之道 — 行为审计层

> 金曰从革，杀伐决断。秋毫必录，功过分明。

**安全哲学**：就像白虎主兵戈、主刑杀，行为审计要记录一切，秋后算账。

**工程实现**：

| 原则 | 说明 | 实现方式 |
|------|------|---------|
| **操作全记录** | 所有操作必须留下痕迹 | 谁、什么时候、做了什么、结果如何 |
| **日志不可篡改** | 审计日志写入后不可修改 | 追加写入，Hash 链校验，异地备份 |
| **异常行为检测** | 偏离正常模式自动告警 | 行为基线学习，异常实时告警 |
| **权限审计** | 定期审计所有权限配置 | 自动发现过度授权、过期权限 |
| **完整可追溯** | 任何问题都能完整复盘 | 执行链路完整记录，支持事后重放 |

**核心信条**：*凡走过，必留下痕迹。凡做过，必可被审计。*

---

## 🧩 八卦八层 · 外御其侮

八卦八层是 OpenTaiji 的外部防御体系，抵御来自系统边界外的各种威胁。

---

### 第五层 ☷ 坤卦 · 地之道 — 多租户隔离层

> 坤厚载物，德合无疆。各安其分，不相侵越。

**安全哲学**：大地承载万物，但万物各有其位。多租户之间必须有不可逾越的边界。

**工程实现**：

| 原则 | 说明 | 实现方式 |
|------|------|---------|
| **数据硬隔离** | 租户数据物理/逻辑隔离 | 独立 Schema / 独立数据库 / 行级安全 |
| **跨租户访问禁止** | 默认禁止跨租户数据访问 | 跨租户访问必须显式授权，全程审计 |
| **配额硬限制** | 每个租户资源有硬上限 | CPU、内存、存储、调用次数，超量自动限流 |
| **计费边界清晰** | 租户消耗精确计量 | 每一次调用都记账，账单可核对可追溯 |
| **租户生命周期管理** | 从创建到销毁完整管控 | 数据清理彻底，无残留，可彻底删除 |

**核心信条**：*租户的就是租户的，一分一毫都不能错，一丝一毫都不能越。*

---

### 第六层 ☶ 艮卦 · 山之道 — 边界隔离层

> 艮为山，静止也。坚不可摧，不可逾越。

**安全哲学**：就像高山矗立，边界就是不可逾越的红线。外部世界再混乱，边界之内永远安全。

**工程实现**：

| 原则 | 说明 | 实现方式 |
|------|------|---------|
| **信任边界明确** | 每一层都有明确的信任边界 | 不信任边界外的任何输入 |
| **白名单优先** | 只允许明确许可的通过 | 默认拒绝，只开放必须的端口/API/权限 |
| **零信任原则** | 即使内部访问也要验证 | 不相信内网，每次调用都验证身份 |
| **网络分区** | 不同安全等级网络隔离 | VPC 划分，安全组，网络 ACL |
| **DMZ 隔离** | 外部服务放在 DMZ 区 | 内外网之间有严格的防火墙策略 |

**核心信条**：*边界之内是家，边界之外是丛林。永远不要相信丛林里的任何东西。*

---

### 第七层 ☵ 坎卦 · 水之道 — 输入验证层

> 坎为水，险陷也。层层过滤，去伪存真。

**安全哲学**：就像水流过层层滤网，所有输入在到达核心逻辑之前，必须经过最严格的验证。

**工程实现**：

| 原则 | 说明 | 实现方式 |
|------|------|---------|
| **所有输入都是恶意的** | 永远不要相信任何输入 | 包括用户输入、API 调用、甚至内部服务返回 |
| **强类型校验** | 类型不对直接拒绝 | JSON Schema，TypeScript 运行时类型检查 |
| **长度限制** | 所有输入必须有长度上限 | 防止溢出攻击，防止拒绝服务 |
| **格式校验** | 格式严格校验 | 邮箱、手机号、URL、特殊字符过滤 |
| **SQL/NoSQL 注入防护** | 所有查询必须参数化 | ORM 参数化查询，禁止字符串拼接 |
| **XSS 防护** | 所有输出自动转义 | 内容安全策略，输入输出双重过滤 |

**核心信条**：*在输入验证上花 1 分钟，在后续排查上省 100 小时。*

---

### 第八层 ☴ 巽卦 · 风之道 — 流量治理层

> 巽为风，入也。疏堵结合，收放自如。

**安全哲学**：就像风无孔不入，流量也是无处不在。不能只堵不疏，要治理要引导。

**工程实现**：

| 原则 | 说明 | 实现方式 |
|------|------|---------|
| **限流熔断** | 保护系统不被冲垮 | 令牌桶、漏桶、熔断降级 |
| **动态流控** | 根据系统负载自动调整 | 负载超过阈值自动限流，恢复后自动放开 |
| **防暴力破解** | 检测并拦截暴力尝试 | 密码错误次数限制，IP 黑名单 |
| **防 DDoS** | 多层防御拒绝服务 | 流量清洗，速率限制，请求排队 |
| **灰度发布** | 新功能灰度放量 | 按比例、按租户、按地区逐步放量 |
| **流量染色** | 关键流量可追踪 | 全链路 Trace ID，端到端可追踪 |

**核心信条**：*系统永远会被攻击，你要做的是确保被攻击的时候还能活着。*

---

### 第九层 ☳ 震卦 · 雷之道 — 网络防御层

> 震为雷，动也。雷霆万钧，邪祟辟易。

**安全哲学**：就像雷霆击碎黑暗，网络防御层要在攻击到达应用之前就把它消灭。

**工程实现**：

| 原则 | 说明 | 实现方式 |
|------|------|---------|
| **WAF 防护** | Web 应用防火墙 | OWASP Top 10 全防护，规则实时更新 |
| **TLS 全站加密** | 所有传输必须加密 | TLS 1.3+，HSTS，证书自动轮换 |
| **证书管理** | 证书生命周期自动化 | 自动申请、自动部署、自动续期 |
| **DNS 安全** | 防止 DNS 劫持和污染 | DNSSEC，多 DNS 服务商备份 |
| **CDN 防护** | 边缘节点安全能力 | CDN WAF，Bot 管理，边缘缓存 |

**核心信条**：*能在网络层解决的攻击，绝不让它到达应用层。*

---

### 第十层 ☲ 离卦 · 火之道 — 身份认证层

> 离为火，明也。明辨身份，真伪立判。

**安全哲学**：就像火焰照亮一切黑暗，身份认证要明辨每一个访问者的真实身份。

**工程实现**：

| 原则 | 说明 | 实现方式 |
|------|------|---------|
| **零信任认证** | 永不信任，永远验证 | 每次请求都验证，不相信会话 |
| **多因素认证** | 关键操作必须多因素验证 | 密码 + 验证码 + 生物特征 |
| **最小权限原则** | 只授予完成任务所需的最小权限 | 权限粒度到 API 级别，定期审计 |
| **权限分离** | 不同角色权限严格分离 | 普通用户、管理员、审计员，三权分立 |
| **JWT 安全** | Token 必须安全使用 | 短有效期，刷新机制，黑名单，加密签名 |
| **OAuth 安全** | 第三方接入安全可控 | 授权码模式，PKCE， scope 最小化 |

**核心信条**：*在身份认证上犯的错，都是根本性的、灾难性的。*

---

### 第十一层 ☱ 兑卦 · 泽之道 — 数据保护层

> 兑为泽，悦也。藏富于泽，密不示人。

**安全哲学**：就像深泽藏宝，敏感数据必须深藏不露，即使被攻破也拿不到明文。

**工程实现**：

| 原则 | 说明 | 实现方式 |
|------|------|---------|
| **静态加密** | 所有敏感数据落盘加密 | AES-256-GCM，密钥管理系统 (KMS) |
| **传输加密** | 所有传输全程加密 | TLS 1.3，中间任何节点不接触明文 |
| **使用加密** | 内存中也是加密状态 | 敏感数据使用时才解密，用完立即清零 |
| **密钥分离** | 密钥和数据分开存储 | 数据在数据库，密钥在 KMS，分开权限 |
| **自动轮换** | 密钥定期自动轮换 | 自动轮换，历史数据可解密，新数据用新密钥 |
| **脱敏展示** | 敏感信息不可完整展示 | 手机号、邮箱、身份证，展示必须脱敏 |
| **删除即销毁** | 删除的数据不可恢复 | 覆写加密，密钥销毁，彻底无法恢复 |

**核心信条**：*假设数据库一定会被拖库，你要做的是拖走了也没用。*

---

### 第十二层 ☰ 乾卦 · 天之道 — 合规审计层

> 乾为天，健也。天行健，君子以自强不息。

**安全哲学**：就像上天监督一切，合规审计层要确保整个系统的每一个角落都符合规范。这是最高的、也是最终的安全保障。

**工程实现**：

| 原则 | 说明 | 实现方式 |
|------|------|---------|
| **合规基线** | 所有操作必须符合合规要求 | GDPR、等保 2.0、SOC 2、ISO 27001 |
| **定期安全审计** | 定期进行全面安全审计 | 每季度内部审计，每年第三方审计 |
| **渗透测试** | 定期模拟真实攻击 | 白帽渗透，红队演练，漏洞修复验证 |
| **漏洞管理** | 完整的漏洞生命周期管理 | 发现 → 评级 → 修复 → 验证 → 关闭 |
| **依赖扫描** | 第三方依赖持续扫描 | SCA 软件成分分析，漏洞及时修复 |
| **安全教育** | 所有人员定期安全培训 | 安全意识，最佳实践，应急演练 |
| **应急响应** | 完整的安全事件响应流程 | 发现 → 遏制 → 根除 → 恢复 → 复盘 |

**核心信条**：*安全是一场没有终点的马拉松，永远不能停下脚步。*

## 🎯 安全核心价值观

### 1. 默认安全 > 事后补救

> 上医治未病，中医治欲病，下医治已病。

所有功能必须默认安全，不能把安全配置丢给用户。安全应该是开箱即用的，不是可选的插件。

### 2. 纵深防御 > 单点防护

> 千里之堤，溃于蚁穴。

不要相信任何一个"绝对安全"的点。要建立多层防御，即使一层被攻破，还有下一层。

### 3. 简洁安全 > 复杂安全

> 大道至简。

安全机制越简单，越容易正确实现，越不容易出错。复杂的安全体系本身就是安全隐患。

### 4. 透明安全 > 模糊安全

> 君子坦荡荡，小人长戚戚。

安全架构应该是公开透明的。不要靠"保密"来获得安全，要靠即使公开了别人也攻不破来获得安全。

### 5. 平衡安全 > 极端安全

> 中庸之为德也，其至矣乎。

安全不是越极端越好。要在安全、便利、性能之间找到平衡点。为了安全把系统搞得完全没法用，那是本末倒置。

## 🛡️ 安全红线

这是 OpenTaiji 项目不可逾越的安全红线，任何人、任何理由都不能违反：

| 红线 | 说明 | 违反后果 |
|------|------|---------|
| **禁止硬编码密钥** | 任何密钥、Token、密码都不能硬编码在代码中 | 代码立即回滚，责任人审查 |
| **禁止明文存储敏感数据** | API Key、密码、Token 必须哈希或加密存储 | 功能立即下线，紧急修复 |
| **禁止关闭安全检查** | 不能为了方便关闭任何安全检查机制 | PR 直接拒绝，不讨论 |
| **禁止绕过权限系统** | 不能用 hack 方式绕过权限校验 | 代码回滚，架构重审 |
| **禁止在日志中打印敏感信息** | 密码、Token、密钥不能出现在日志中 | 日志立即清理，漏洞紧急修复 |
| **禁止用户输入拼接 SQL** | 所有数据库查询必须参数化 | PR 直接打回，安全培训 |

## 🚨 报告安全漏洞

如果你发现了安全漏洞，请**不要**公开在 GitHub Issue 中。请发送邮件至：

**security@opentaiji.dev**

我们会在 24 小时内回复，确认漏洞后会立即修复，并在修复完成后公开致谢。

### 漏洞披露政策

1. 发现漏洞后请第一时间私下通知我们
2. 请给我们合理的修复时间，不要在修复完成前公开漏洞细节
3. 不要利用漏洞进行任何破坏性操作
4. 不要泄露或访问任何用户数据

符合政策的漏洞报告，我们会将你的名字加入安全贡献者荣誉榜。
