## 🚀 快速开始

### 环境要求

- Node.js 20+ 
- TypeScript 5.0+
- npm 10+
- 推荐硬件：16核CPU / 32GB内存（极限压测建议32核/128GB）

### 安装依赖

```bash
cd /root/.openclaw/workspace
npm install
```

### 运行所有测试

```bash
# 完整生产级测试套件（推荐）
npx ts-node tests/production/run-all-tests.ts

# 或使用快捷方式
npm run test:production
```

### 运行单个测试模块

```bash
# 只运行功能完整性测试
npx jest tests/production/core-engine.test.ts --verbose
npx jest tests/production/wfgy-verification.test.ts --verbose
npx jest tests/production/memory-system.test.ts --verbose

# 只运行压力测试
npx ts-node tests/production/stress-test-runner.ts

# 只运行代码质量检查
npx ts-node tests/production/code-quality-check.ts
```

## 🧪 测试详细说明

### 1. 核心引擎模块测试

#### 测试用例1：Actor生命周期管理
- **操作**：批量创建1500个智能体，模拟运行30天后批量终止
- **期望**：初始化成功率100%，终止后无僵尸进程残留
- **验证**：检查进程列表和内存占用曲线

#### 测试用例2：邮箱驱动消息传递
- **操作**：创建50个智能体形成消息链，发送1000条消息
- **期望**：所有消息在5秒内送达，无消息丢失、无乱序
- **验证**：记录每条消息的发送时间和接收时间，计算P95延迟

#### 测试用例3：监督树自愈机制
- **操作**：模拟核心智能体进程崩溃
- **期望**：5秒内检测到故障，10秒内完成重启，任务账本恢复未完成工作
- **验证**：监控日志中的故障检测事件和重启事件时间戳

#### 测试用例4：任务账本持久化与恢复
- **操作**：在智能体处理审批任务中途强制终止进程，5秒后重启
- **期望**：账本数据完整无损，任务从断点继续执行，不重复不遗漏
- **验证**：比对终止前后的任务状态和数据一致性

#### 测试用例5：上下文自组装能力
- **操作**：让智能体参与3轮独立对话，分别获取信息片段，第4轮对话中自动关联
- **期望**：跨对话信息关联成功率≥85%
- **验证**：检查Memory图谱中的关联边数量和质量

### 2. WFGY防幻觉系统测试

#### 测试用例1：合规性规则校验
- **输入**：生成包含违规排放建议的文本（如"COD排放可放宽至200mg/L"）
- **期望**：WFGY正确识别违规，valid=false，输出违规详情
- **验证**：检查WFGYVerificationResult.valid字段和violatedRules数组

#### 测试用例2：多路径自一致性
- **输入**：对同一问题进行5次独立采样
- **期望**：一致性≥70%时输出共识结论，否则声明"无法确定"
- **验证**：计算5次输出的语义相似度

#### 测试用例3：知识溯源索引
- **输入**：要求智能体引用政策法规回答"VOCs排放标准"
- **期望**：每条答复包含可追溯的法规编号和条款
- **验证**：检查输出中是否包含"依据：《XX标准》第X条"

#### 测试用例4：幻觉风险检测
- **输入**：询问超出知识库范围的问题（如"某企业明年排污量预测"）
- **期望**：输出置信度评分，低于0.6时主动声明"我不确定"
- **验证**：检查输出的confidence字段和拒答行为

#### 测试用例5：13种失败模式覆盖测试
- **覆盖**：数值虚构、政策编造、时间穿越、实体不存在、因果倒置、
          单位错误、来源伪造、数据矛盾、过度承诺、责任推卸、
          含糊其辞、引用失效、逻辑断裂
- **期望**：每种模式的检出率≥90%

### 4. 极限压力测试

#### 4.1 智能体并发极限测试

**测试场景**：工业园区智慧环保管家（1100+智能体同时运行）
- 20个核心管理智能体（局长、监察大队、监控中心、审批科、应急科、综合科）
- 1080个企业环保管家智能体（每家企业1个）

**梯度加压**：
```
100 → 200 → 300 → 500 → 800 → 1100 → 1500
每个梯度运行30分钟
```

**拐点检测**：当P95延迟陡增（>2倍基线值）或错误率>5%时，记录该并发量为系统极限

**期望结果**：
- 最大并发智能体数量≥1100
- P95响应延迟<5秒（在80%极限负载下）
- 智能体初始化成功率100%
- 消息传递成功率≥99.9%

---

#### 4.2 消息风暴压力测试

**测试场景**：模拟突发事件（如园区化学品泄漏），短时间产生大量消息传播
- 消息传播链：企业→监控中心→应急科→局长→监察大队→周边企业→园区管理部门
- 预期消息量：5000+条/分钟

**期望结果**：
- 消息风暴期间系统不崩溃
- 关键节点消息优先处理，非关键消息可延迟
- 峰值过后30秒内，邮箱队列恢复至正常水平
- 无消息丢失

---

#### 4.3 业务场景混合压测

**场景配置**：

| 场景 | 负载占比 | 并发数 |
|------|---------|--------|
| 智能监控预警 | 40% | 400 |
| 智能执法辅助 | 20% | 200 |
| 智能溯源分析 | 15% | 150 |
| 智能审批辅助 | 10% | 100 |
| 智能应急指挥 | 5% | 50 |
| 智能报告生成 | 10% | 100 |

**期望结果**：
- 所有场景综合成功率≥99%
- 高优先级场景（预警、应急）响应延迟<3秒
- 各场景之间不相互干扰，无资源争抢导致的死锁

#### 4.4 长时稳定性测试

**测试配置**：
- 并发智能体数量：1100个（拐点负载的80%）
- 业务负载：按上述比例持续运行6大业务场景

**期望结果**：

- 梦境合成成功执行3次，知识图谱规模持续增长
- 错误日志数量无异常增长趋势

## 🔧 故障排查

### 常见问题

**Q: Jest测试报错找不到模块？**
> A: 确保已运行 `npm install` 和 `npm run build` 编译TypeScript文件。

**Q: 压力测试运行很慢？**
> A: 检查机器配置，建议至少16核32GB；可在`stress-test-runner.ts`中调整`durationPerStepMinutes`缩短测试时间。

**Q: 测试结果文件在哪里？**
> A: 所有结果保存在 `stress-test-results/` 目录下，JSON格式的原始数据和Markdown格式的报告都会生成。

**Q: 如何自定义测试参数？**
> A: 修改对应测试文件中的配置常量，例如在`stress-test-runner.ts`中修改`StressTestConfig`。
