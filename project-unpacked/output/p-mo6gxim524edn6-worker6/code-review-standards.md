## 1. 概述

OpenTaiji 作为开源防幻觉 AI 项目，代码质量是项目可靠性和可扩展性的基石。作为技术评审员，我们的使命是：

> **教练式指导，不是挑刺，是帮助团队提升代码质量，提前识别风险，共享最佳实践。**

本标准定义了代码评审的完整框架，涵盖架构一致性、命名规范、错误处理、性能和安全五个核心维度，并针对 OpenTaiji 项目结构做了适配。

## 2. 代码评审规范

### 2.1 评审目标

- 保证代码质量一致性
- 提前识别潜在风险（安全漏洞、性能隐患、架构腐化）
- 确保代码符合项目架构设计
- 分享最佳实践，帮助团队共同进步
- 保证代码可维护性、可读性、可扩展性

### 2.2 评审范围

- 所有新增功能代码
- 所有修改现有功能的代码
- 所有重构代码
- 配置文件变更
- 基础设施代码（Dockerfile、CI配置、部署脚本）
- 测试用例代码

### 2.3 评审流程

```
开发者提交 Pull Request → 自动化检查（lint/test/build） → 技术评审 → 修改 → 批准/拒绝 → 合并
          ↑                          ↓
          └──────────────────────────┘
```

**评审时机**：代码完成开发，自动化测试通过后，进行代码评审。

### 2.4 评审原则

1. **对事不对人**：评审代码，不是评审人
2. **给出明确建议**：每条意见附带改进建议和参考范例
3. **分级优先级**：区分必须改 / 建议改 / 可选优化
4. **尊重既有风格**：与项目既有风格一致时不强制个人偏好让位于约定
5. **简洁明了**：一句话说清楚问题、原因和改进方向

## 3. 架构一致性标准

### 3.1 项目架构分层

OpenTaiji 当前项目结构：

```
src/
├── modules/             # 功能模块层
│   ├── determinism/    # 防幻觉确定性系统
│   │   ├── interfaces/ # 接口定义
│   │   ├── WFGYVerifier.ts
│   │   ├── SelfConsistencyChecker.ts
│   │   ├── SourceTracer.ts
│   │   ├── HallucinationDetector.ts
│   │   └── DeterminismSystem.ts
│   ├── [other modules]/ # 未来新增模块
├── cli/                 # 命令行入口
├── utils/              # 通用工具
└── types/              # 全局类型定义
```

**分层规则**：

| 层级 | 职责 | 依赖规则 |
|------|------|----------|
| interfaces | 定义抽象接口 | 不依赖任何实现层 |
| 模块组件 | 单一功能组件实现 | 只依赖接口，不依赖其他组件实现（除非明确组合） |
| 系统入口 | 组合多个组件提供完整功能 | 依赖所有组件，可以注入引用 |
| utils | 通用工具函数 | 可被所有层依赖，不反向依赖业务 |

### 3.2 模块划分原则 (OpenTaiji 特定)

OpenTaiji 按**功能职责**划分模块，每个模块一个明确的业务能力：

- `determinism` 模块：防幻觉确定性验证，包含多个单一职责组件
  - `WFGYVerifier`: 只负责符号层规则验证
  - `SelfConsistencyChecker`: 只负责多采样一致性检查
  - `SourceTracer`: 只负责知识来源溯源
  - `HallucinationDetector`: 综合评分幻觉风险
  - `DeterminismSystem`: 组合所有组件，提供统一入口

**原则**:
- 单一职责：一个组件只做一件事
- 组合优于继承：通过组合多个组件完成复杂功能
- 接口与实现分离：接口在 `interfaces`，实现在上层
- 可独立测试：每个组件都可以单独单元测试

### 3.3 依赖关系规则

**检查要点**:
- [ ] 是否存在循环依赖
- [ ] 是否存在不必要的依赖
- [ ] 是否违反分层依赖规则（接口不依赖实现，实现不依赖入口）
- [ ] 第三方依赖引入是否必要
- [ ] 依赖版本是否在根 `package.json` 统一管理

**禁止**:
```typescript
// 禁止循环依赖：A 导入 B，B 导入 A
// 禁止反向依赖：实现导入组合入口类
// 禁止跳过接口直接依赖具体实现（除非是组合场景）
```

### 3.4 一致性检查清单

- [ ] 代码是否符合项目既定的架构分层
- [ ] 是否存在循环依赖
- [ ] 依赖方向是否正确（接口 → 实现 → 组合，不反向）
- [ ] 模块/组件职责是否单一清晰
- [ ] 是否存在上帝组件（一个文件做太多职责，超过 500 行需要警惕）
- [ ] 接口抽象是否合理
- [ ] 是否有不必要的抽象
- [ ] 是否遵循项目已有的目录结构约定
- [ ] 新增模块是否合理划分边界
- [ ] 是否存在代码重复（是否应该抽取复用到 utils）
- [ ] 组件之间是否正确解耦，是否可以独立测试

## 4. 命名规范

### 4.1 通用原则

1. **见名知意**：命名要表达意图，不需要注释就能理解
2. **避免缩写**：除非是广为接受的缩写（HTTP、URL、ID、AI 等）
3. **长度适当**：不要过短（含义模糊），不要过长（冗余）
4. **一致性**：相同概念在整个项目中保持相同命名

**坏例子**:
```typescript
// 坏：含义模糊
let u: User;
function getData() {}

// 好：清晰表达
let currentUser: User;
function getUserByID(userID: string): Promise<User> {}
```

### 4.2 TypeScript/JavaScript 特定规范

| 元素 | 命名规则 | 范例 |
|------|----------|------|
| 文件 | kebab-case | `wfgy-verifier.ts` 不是 `WFGYVerifier.ts`（当前项目约定，保持一致） |
| 类/接口/类型 | PascalCase | `WFGYVerifier`, `IDeterminismSystem` |
| 函数/变量/方法 | camelCase | `verifySymbols`, `extractSymbols` |
| 常量 | UPPER_SNAKE_CASE | `MAX_SAMPLES`, `DEFAULT_THRESHOLD` |
| 私有成员 | 私有前缀 `_` | `private _symbolRegex: RegExp | null` |
| 接口 | 前缀 `I` | `IDeterminismSystem` |

### 4.3 命名检查清单

- [ ] 命名是否表达了做什么/是什么，而不是如何做
- [ ] 是否存在含义模糊的单字符命名（除外循环计数器等局部变量）
- [ ] 是否存在不一致命名（同一个概念多个命名）
- [ ] 是否存在错误大小写误用
- [ ] 布尔变量命名是否清晰表达布尔含义（`isXxx`, `hasXxx`, `canXxx`）
- [ ] 函数命名是否清晰表达动作+对象（`getUser`, `createUser`）
- [ ] 是否存在不必要下划线/大小写不一致
- [ ] 文件名是否遵循项目约定（kebab-case）

## 5. 错误处理完整性检查

### 5.1 错误处理原则

1. **不吞错误**：所有错误都必须被处理或传递
2. **错误信息有上下文**：包装错误添加上下文便于排查
3. **不抛异常控制流程**：使用返回值代替异常控制业务流程
4. **尽早失败**：非法输入尽早返回错误
5. **统一错误处理**：遵循项目统一的错误处理模式

### 5.2 检查要点

**常见问题**:

1. **吞错误**:
```typescript
// 错误：忽略了错误
try {
  JSON.parse(body);
} catch (e) {
  // 什么都不做
}

// 正确：处理错误返回明确结果
try {
  return JSON.parse(body);
} catch (e) {
  return {
    valid: false,
    error: `Failed to parse JSON: ${(e as Error).message}`
  };
}
```

2. **缺失边界检查**:
- 数组/越界检查
- `null`/`undefined` 检查
- 空字符串检查
- 空数组检查

3. **空输入没有防御**:
```typescript
// 不好：调用方传 null 会导致运行时错误
function verify(content: string): Result {
  return regex.test(content);
}
// 好：
function verify(content: string | null | undefined): Result {
  if (content == null) {
    return { valid: false, error: 'Content cannot be null' };
  }
  // ...
}
```

4. **异步错误没有捕获**:
```typescript
// 不好：没有 catch，Promise  rejection 会抛出未捕获异常
async function fetchData(): Promise<Data> {
  const resp = await fetch(url);
  return resp.json();
}
// 好：
async function fetchData(): Promise<Result<Data, Error>> {
  try {
    const resp = await fetch(url);
    if (!resp.ok) {
      return { ok: false, error: new Error(`HTTP ${resp.status}`) };
    }
    return { ok: true, data: await resp.json() };
  } catch (e) {
    return { ok: false, error: e as Error };
  }
}
```

### 5.3 错误处理检查清单

- [ ] 所有可能抛出的异步调用是否 `try/catch`？
- [ ] 是否存在吞错误（忽略 catch 不处理）的情况？
- [ ] 边界条件（输入、输出、参数、`null`/`undefined`、空）都检查了吗？
- [ ] 错误是否包含上下文信息便于排查？
- [ ] 错误处理后程序执行流是否正确退出？
- [ ] 是否滥用 `throw` 进行业务控制流？
- [ ] 资源清理（关闭文件/连接）是否处理错误？
- [ ] 异步 Promise 是否处理 rejection？

## 6. 性能隐患检查要点

### 6.1 常见性能问题 (AI 应用特定)

**内存相关**:

1. **大对象重复分配**:
```typescript
// 在循环中频繁创建大对象，应该复用
for (const item of items) {
  const buf = new Array(10000); // 每次都分配大内存
}
```

2. **数组/Set 预分配**:
- 知道长度，应该预分配容量
- 动态增长会多次重新分配内存

3. **不必要拷贝**:
- 字符串不必要切片产生拷贝
- 大对象传值而不是传引用

**算法复杂度**:

1. **O(n^2) 算法**:
- 符号匹配使用 O(n*m) 算法，知识库大了变慢
- 优化为一次扫描 O(n) 算法（如 WFGY 已经优化）

2. **N+1 查询问题**:
- 循环中查询知识库，应该批量查询

3. **正则表达式陷阱**:
- 避免复杂嵌套正则，可能导致灾难性回溯
- 使用简单明确的正则

**缓存相关**:

- 热点数据没有缓存，重复计算
- 缓存不失效导致内存泄漏
- 缓存击穿没有保护

**AI 特定**:

- LLM 调用重复不必要，应该缓存结果
-  embedding 重复计算，应该缓存
- 大文本全量加载到内存，应该流式处理

### 6.2 性能检查清单

- [ ] 循环内部是否存在不必要的分配、IO 操作？
- [ ] 数组、map 是否预分配了容量？
- [ ] 是否存在 N+1 查询问题？
- [ ] 是否存在大对象在循环中分配？
- [ ] 是否存在不必要的拷贝？
- [ ] 锁的粒度是否过大？
- [ ] 是否存在协程/Promise 泄漏风险？
- [ ] 是否缓存了可缓存的计算结果？
- [ ] 缓存是否有过期/清理策略避免内存泄漏？
- [ ] 大文本计算是否使用流式处理而不是全量加载到内存？
- [ ] 正则表达式是否存在灾难性回溯风险？
- [ ] LLM API 调用是否有缓存减少重复调用？

## 7. 安全漏洞检查要点

### 7.1 OWASP Top 10 重点检查

**A01:2021 注入**
- 命令注入：系统命令是否拼接用户输入？
- 正则注入：用户可控的正则是否编译执行？

**A02:2021 身份认证**
- （当前项目不涉及，但需要注意如果开放 API 需要）

**A03:2021 敏感数据暴露**
- 是否日志打印了用户敏感数据？
- 知识库中敏感数据是否加密存储？

**A05:2021 访问控制错误**
- （当前项目不涉及，本地部署为主）

**A06:2021 不安全配置**
- 依赖是否存在已知 CVE 漏洞？
- 是否暴露了调试信息给用户？

**A07:2021 XSS 跨站脚本**

**A08:2021 反序列化漏洞**

**依赖安全**:
- 第三方 AI SDK 是否可信？

### 7.3 安全检查清单

- [ ] 所有用户可控输入是否校验/过滤？

## 9. 评审结论分级

### 必须改（Block）：

- 存在安全漏洞
- 存在严重性能隐患（OOM，O(n^2) 在热点路径）
- 违反架构一致性规则（循环依赖，反向依赖）
- 吞错误严重问题
- 影响功能正确性严重 Bug

### 建议改（Comment）：

- 命名不清晰可改进
- 可维护性问题（文件过大，职责不清晰）
- 性能可优化
- 架构一致性小偏差不影响整体

### 可选优化（Suggestion）：

- 代码可以更好改进点，但不影响当前质量
- 个人风格偏好，不违反规则
- 未来重构建议
- 未来优化空间
